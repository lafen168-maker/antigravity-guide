核心概念與設計原理

深入了解 Antigravity 如何擺脫傳統 AI 助手框架，邁向自主開發時代。

Agent-First (Agent 優先)

傳統 AI 只是「自動補全程式碼」或在「側邊欄進行聊天」。Antigravity 則是 Agent-First 機制：你只需給出一個高階目標，它便會主動搜尋、設計、建立、修改代碼、執行命令並完成測試，過程中無需你一行行地指導。

三介面協同運作

將 編輯器 (Editor)、管理器 (Agent Manager) 與 瀏覽器 (Browser) 整合在同一個 Actuation 迴圈中。Agent 可以像人類開發者一樣，編寫代碼後立刻打開瀏覽器測試，並根據截圖或報錯進行自我修復 (Self-healing)。

Artifacts 信任機制

不同於黑盒子運作，Antigravity 將重要里程碑產出為 Artifacts (交付產物)，包含實作計畫書、任務清單與最後的 Walkthrough 報告。開發者能在 Agent 執行前進行評估與核准，建立了高度的掌控性與信任感。

編輯器：精準安全的代碼編寫

基於現代 IDE 架構（如 VS Code API），Agent 可直接讀取專案的目錄結構，並以「非破壞性」方式對檔案進行單一或多處非連續性的代碼替換。

  • Replace File Content: 精確比對目標區塊代碼並替換，防止全檔覆寫錯誤。
  • Multi-Replace File Content: 跨行、多區塊一次性修改，大幅縮短執行時間。
  • Grep Search: 支援正規表示法搜尋，快速定位整個專案中的函式或變數。

管理器：多任務調度中心

在處理大型專案時，單一 AI 無法在單一上下文（Context）中完美處理所有事情。管理器允許您或主 Agent 動態產生多個分工明確的子 Agent，進行異步並行開發。

  • Define Subagent: 賦予子 Agent 專屬的 System Prompt 與工具集（如唯讀研究、代碼寫入）。
  • Invoke & Send Message: 主子 Agent 之間可傳遞參數、指派子任務，並將結果回傳主執行流。
  • Kill & Status Control: 隨時停止特定分支的執行，保護運算資源不浪費。

瀏覽器：最終成果的「視覺驗證」

程式碼編譯成功並不代表介面完全正確。Antigravity 的瀏覽器表面（Browser Surface）提供 E2E 視覺驗證機制，讓 Agent 模擬真實使用者互動。

  • Chrome Automation: Agent 可模擬點擊按鈕、填寫輸入框、滾動頁面等動作。
  • 視覺截圖與錄影: 將測試過程產出為截圖與影片 Artifact，做為給教師/使用者的交付證據。
  • 自我修正 (Self-healing): 當瀏覽器報錯（Console Error）或畫面渲染異常時，Agent 會自動讀取錯誤並修正代碼，直到驗證通過。

教師教學引導與使用步驟

如何在教學或學生練習中導入 Antigravity？遵循以下 5 大標準步驟。

STEP 01

指派目標與情境設定 (Targeting)

教師或學生向 Antigravity 提出明確的目標任務。建議使用「角色 (Role) + 任務 (Task) + 限制條件 (Constraint)」的格式。

教學範例：「你是一位前端工程師，請為我製作一個班級出缺席登記系統的單頁網頁，需要支援離線暫存與響應式排版，並且以繁體中文顯示。」
STEP 02

自主研究與實作計畫 (Research & Plan)

Agent 收到任務後，會先搜尋工作區目錄、探查依賴環境（Research）。接著，Agent 必須產出一個 implementation_plan.md（實作計畫書）存於 Artifact 目錄中。計畫書內會詳列修改檔案清單、所用技術、未解問題與驗證計畫。

STEP 03

計畫審查與人工批准 (Approval)

這是教學上的核心控管點！ Agent 產生計畫後會暫停執行，並向使用者發出審查請求。教師或學生在此時可以：

  • 詳細閱讀計畫，指出邏輯盲點，藉此培養學生的程式架構評估能力。
  • 在計畫書中留言反饋，Agent 會自動讀取並修正計畫。
  • 點擊「核准 (Approve)」或在對話框中回覆「同意」，Agent 才會進入代碼寫入階段。
STEP 04

自動化執行與任務追蹤 (Execution)

獲得批准後，Agent 會建立一個 task.md（待辦工作清單），並開始調用編輯器與終端機工具進行開發。開發過程中，Agent 會將進行中的任務標記為 [/]，完成後標記為 [x]，以便使用者實時觀看開發進度。

STEP 05

視覺驗證與交付報告 (Verification)

開發完成後，Agent 會自動啟動虛擬瀏覽器或運行測試腳本。驗證成功後，Agent 會產出一份 walkthrough.md 報告，詳細說明修改細節，並附上自動測試的瀏覽器截圖，作為學生向老師交作業或成果展示的依據。

教師專用 Prompt 範本與範本

可以直接複製並在 Antigravity 中運行的情境範本，降低教學導入門檻。

場景說明

當老師需要為課堂製作一個輔助演示工具（例如：質數篩選器、物理碰撞模擬器、單字卡抽卡器）時，可以使用此範本讓 Agent 快速產出具有高質感的單頁網頁教材。

Prompt 內容 
請扮演前端教學網頁開發專家。我需要一個供國小五年級學生學習「質數與合數」的動態網頁教具。

功能要求：
1. 畫面以 1 到 100 的數字網格展示。
2. 學生點擊任何數字時，會以動畫展示該數字的「因數分解」過程。
3. 提供一個「埃拉托斯特尼篩法 (Sieve of Eratosthenes)」自動演示按鈕，點擊後會逐步（每次間隔1秒）將 2, 3, 5, 7 的倍數用不同顏色篩去，最後留下的質數會發光。
4. 介面需為繁體中文，採用明亮、活潑、對比鮮明的卡通科技感風格（如黃、橘、天藍色搭配）。
5. 以純 HTML/CSS/JavaScript 撰寫，整合在單一 index.html 中，以方便我直接分發給學生開啟。

請先進行研究並提供實作計畫，等待我批准後再開始撰寫。

場景說明

學生交來的程式碼作業可能雜亂無章、缺乏註解、或效能低下。老師可以指派 Agent 讀取該作業，進行代碼結構重構、補齊繁體中文註解，並產出重構前後的對比分析，作為教學範例。

Prompt 內容 
請分析工作區中學生的 JavaScript 專案。這是一個簡易的購物車計算邏輯。

任務目標：
1. 找出程式碼中潛在的 Bug（如浮點數計算精度錯誤、全域變數污染）。
2. 重構程式碼：將冗長且重複的巢狀 if-else 邏輯改為更具可讀性的 Array 方法（如 map, filter, reduce）。
3. 程式碼中必須為每個函式補上詳細的繁體中文 JSDoc 註解，說明參數與回傳值。
4. 保持原有的外部介面 API 不變，確保不會損壞其他連接的元件。

請提供實作計畫（含重構前後的架構對比想法），經我確認後執行，並在完成後提供 walkthrough.md 說明修改原因與測試結果。

場景說明

在教授網頁開發時，讓學生手動測試響應式網頁非常耗時。本範本指派 Agent 自動開啟瀏覽器，針對特定網頁進行多解析度（RWD）截圖驗證，自動檢查是否有文字溢出或按鈕重疊，並產出圖文並茂的測試報告。

Prompt 內容 
我需要對目前工作區運行的本機網頁 (http://localhost:3000) 進行自動化響應式介面測試。

測試要求：
1. 啟動瀏覽器並導覽至該網頁。
2. 分別在以下三種螢幕解析度下進行測試：
   - 桌上型電腦: 1920x1080
   - 平板電腦: 768x1024
   - 行動裝置: 375x812
3. 在每種解析度下，請將瀏覽器視窗調整至對應尺寸，並分別擷取「首頁英雄區」與「聯絡表單區」的完整截圖。
4. 檢查行動裝置解析度下，頂部導覽列是否已正確收納為漢堡選單 (Hamburger Menu)，按鈕是否可被正常點選。
5. 將測試產出的 6 張截圖打包，並在 `walkthrough.md` 中詳細列出在各解析度下的排版檢查結果（是否通過 RWD 檢驗）。

請在實作計畫中詳述測試步驟，獲得批准後開始進行瀏覽器自動化驗證。

內部設定與斜線指令手冊

掌握進階參數與快捷指令，讓 Antigravity 的運作更精準、更符合您的教學要求。

常用斜線指令 (Slash Commands)

在與 Antigravity 對話時，您可以推薦或直接輸入以下快捷指令，快速調用特定工作流：

/goal
終極任務模式：當你需要進行一個超長、跨夜、極為複雜的專案時使用。Agent 會被設定為「極致細心」，在完全達成高階目標前不會輕易停止或向用戶回報。
/schedule
排程任務：可設定一次性倒數計時器或 Recurring Cron（循環任務）。例如：每隔 5 分鐘自動運行一次健康檢查，或在一小時後提醒檢查代碼打包進度。
/browser
瀏覽器優先模式：指示 Agent 將任務重點放在 Web 瀏覽、網路爬蟲或網頁端 UI 驗證，自動配備所需的 Chrome 網頁互動驅動。
/grill-me
互動式拷問/對齊模式：在執行複雜任務前，讓 Agent 對您進行一系列的選擇題或問答，釐清不明確的設計細節、資料庫欄位或 UI 風格，避免做白工。

核心工具說明與運作參數

Antigravity 透過限制沙盒環境與明確的 API 工具，保障程式碼的修改品質與系統安全：

replace_file_content
單一連續區塊修改：必須提供精確的 StartLine、EndLine 與與原檔案 100% 吻合的 TargetContent。最安全的修改手段。
multi_replace_file_content
多非連續區塊修改：當需要同時修改同一個檔案中的多個不相鄰區塊時使用。能將多個 ReplacementChunk 打包，防止多次寫入產生的狀態衝突。
ask_permission
權限授權申請：當 Agent 需要讀寫受保護的目錄，或執行涉及敏感權限的命令時，會觸發此工具，向使用者申請單次或特定規則的豁免權限。
define_subagent
定義子 Agent 規格：包含子 Agent 的名稱、職責系統提示（System Prompt），以及是否賦予其寫入權限 (enable_write_tools) 或調用 MCP 伺服器權限。
